Feature Description

Allow users to manage roles and permissions in a way that supports their workflows and content-access policies. Provide users with an easy way to define custom roles and fine-tune permissions without intervention by O'Reilly staff.

The following roles are likely to be fairly common among publishers, but individual permissions may vary from project to project or company to company. I think it's likely that publishers will want to have multiple versions of each: for example, they may want a type of Author role that has fairly limited permissions, while another Author role could be granted a wider range of powers, such as creating a new project). The permissions module should give them this flexibility.

  • Author
  • Reviewer
  • Editor
  • Production 
  • Production Manager
  • Copyeditor
  • Indexer
  • Proofreader
  • Publisher
  • <custom>

I suggest providing a starter solutions with at least some of these roles in place, with reasonable permissions for each, to get them started.

A few examples: Author may be an all-powerful role in one company, but fairly weak in another. If a company employs a mix of in-house and external copyeditors, they may wish to define permissions two copyeditor roles.

For the way O'Reilly works, the roles Copyeditor, Indexer, and Proofreader would have the same permissions. The arguments I can offer for not simplifying this scheme to a more generic role would be the quick visibility everyone in a project would have through meaningful role names and the possibility that other publishers won't be set up the same way with respect to vendors.

The following are permission categories that I think are relevant. We should flesh this list out as part of the initial discussions before development:

  • create a new project
  • modify any project in a group
  • delete own projects
  • delete any project in a group
  • download own project
  • download any project in a group
  • create new branch in own project
  • create new branch in any project in a group
  • merge changes in own project
  • merge changes in any project in a group
  • view own project metadata/statistics
  • view metadata/statists of any project in a group
  • invite a new user to own project
  • invite new user to any project in a group
  • invite a user to a group
  • remove collaborator from any project in a group
  • remove collaborator from a group
  • edit build settings (separate from changing a theme)
  • change theme (separate from setting order of chapters in build)
  • modify css 
  • access git url

Breaking down permissions in this way could give us finer control than more simple read/write based permissions, although at the cost of complexity.

Success Criteria

Users are able to manage their own permissions and roles after an initial introduction to the system and intervention by O'Reilly staff isn't needed. 

Maximum Level of Development Effort

My guess is medium. I need input from the team.
